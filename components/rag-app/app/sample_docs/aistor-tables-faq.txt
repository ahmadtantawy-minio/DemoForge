AIStor Tables: Built-In Apache Iceberg REST Catalog for MinIO

AIStor Tables is MinIO's built-in Apache Iceberg V3 REST Catalog, integrated directly into the MinIO server. It provides full lakehouse capabilities — schema evolution, time travel, partition evolution, and multi-engine query access — without requiring any external catalog service. There is no need to deploy or manage a separate Hive Metastore, AWS Glue, or any other third-party catalog. The catalog is simply part of MinIO itself.

This architectural decision is one of the most significant differences between AIStor Tables and competing solutions. Traditional lakehouse setups require a storage layer (S3 or HDFS), a catalog layer (Hive Metastore or AWS Glue), and a query engine, each deployed and operated independently. AIStor Tables collapses the storage and catalog layers into a single system. When you write an Iceberg table through AIStor Tables, the catalog metadata is managed by MinIO directly, stored alongside the data in the same object store, and served through a standard Iceberg REST Catalog API endpoint that any compatible query engine can speak.

The Iceberg REST Catalog API is the interface that makes AIStor Tables interoperable. Apache Iceberg defines a REST specification for catalog operations — creating namespaces, registering tables, loading table metadata, committing transactions, and listing tables. AIStor Tables implements this specification fully, which means any engine that supports Iceberg REST catalogs can connect to it without custom configuration or plugins. Spark, Trino, Dremio, StarRocks, and other engines all support the Iceberg REST Catalog API natively.

Comparing AIStor Tables to AWS S3 Tables is instructive. AWS S3 Tables, announced in late 2024, offers managed Iceberg tables within S3, but it is tightly coupled to the AWS ecosystem. It requires AWS Glue as the catalog backend, operates only within specific AWS regions, and introduces per-request pricing on top of standard S3 costs. AIStor Tables, by contrast, runs anywhere MinIO runs — on-premises, in any cloud, or at the edge — and the catalog functionality is included as part of the AIStor product rather than billed separately. Organizations that need data sovereignty, air-gapped environments, or multi-cloud flexibility find AIStor Tables a more flexible option.

Schema evolution is a core Iceberg capability that AIStor Tables exposes fully. As data formats change over time — new columns are added, types are widened, or columns are renamed — Iceberg records these changes in the table metadata rather than rewriting historical data. A query reading data written two years ago and data written today sees a unified schema, with Iceberg handling the translation transparently. AIStor Tables preserves the complete schema history for every table, making it possible to understand exactly what the schema looked like at any point in time.

Time travel is another foundational feature. Every write to an Iceberg table creates a new snapshot — a point-in-time view of the table's contents. AIStor Tables retains these snapshots according to configurable retention policies. Any query engine connected to AIStor Tables can specify a snapshot ID or a timestamp and read the table exactly as it existed at that moment. This is invaluable for debugging data quality issues, reproducing historical results, or auditing changes to critical datasets.

Partition evolution allows the partitioning strategy of a table to change over time without rewriting existing data. A table that was originally partitioned by month can be repartitioned by day for newer data, with the query engine handling both partition layouts seamlessly. AIStor Tables tracks the partition spec history as part of the table metadata, ensuring that queries spanning partition evolution boundaries return correct results without manual intervention.

The lakehouse architecture enabled by AIStor Tables eliminates the traditional trade-off between data lake flexibility and data warehouse query performance. In a classic data lake, raw data sits in files on object storage and is accessible to any tool, but query performance is limited by the lack of indexing and the overhead of scanning large files. In a classic data warehouse, data is optimized for query performance but locked into a proprietary format and separated from the raw source. AIStor Tables enables a third path: open table format (Apache Iceberg V3) stored directly on MinIO object storage, queryable by any compatible engine at high performance, with no data movement or format conversion required.

For teams already using MinIO as their object storage layer, adopting AIStor Tables requires no new infrastructure. The REST Catalog endpoint is available on the same MinIO server, authenticated through the same credential system, and backed by the same durable, erasure-coded storage that protects all other data in the cluster. This means organizations can build a complete, production-grade lakehouse without adding operational complexity — no additional services to deploy, no additional systems to monitor, and no additional failure domains to manage.

AIStor Tables is the answer for organizations asking whether they can run a modern lakehouse without a cloud-managed catalog service. The answer is yes: AIStor Tables provides everything a catalog needs to be, embedded directly in the storage system where the data already lives.
