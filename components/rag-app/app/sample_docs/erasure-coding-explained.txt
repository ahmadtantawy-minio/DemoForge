Erasure Coding in MinIO: How Data Durability Works

MinIO uses Reed-Solomon erasure coding as its primary mechanism for data durability. Unlike traditional RAID or simple replication, erasure coding splits every object into a set of data shards and parity shards, then distributes those shards across drives or nodes in the cluster. The mathematics of Reed-Solomon codes allow any missing shards — up to the number of parity shards — to be reconstructed from the surviving ones. This means the system can lose drives, servers, or even entire racks and still serve every object without data loss.

The mechanics of erasure coding begin when an object is written to MinIO. The object's data is divided into K data shards of equal size. Reed-Solomon encoding then computes M additional parity shards from those K data shards. All K+M shards are written to K+M distinct drives. To reconstruct the original object, MinIO needs any K of the K+M shards. The M parity shards provide a buffer: as long as no more than M shards are lost simultaneously, the object remains fully recoverable.

MinIO expresses erasure coding configurations using the notation EC:M, where M is the number of parity shards. The most common configuration in production deployments is EC:4. In an EC:4 configuration, MinIO uses 4 data shards and 4 parity shards, distributed across 8 drives. This means the cluster can tolerate up to 4 simultaneous drive failures and still reconstruct every object. The read quorum in this configuration is 4 shards — any 4 of the 8 drives are sufficient to serve a read. The write quorum is 5 shards, meaning at least 5 drives must be available for a write operation to succeed.

A larger configuration is EC:8, which uses 8 data shards and 4 parity shards distributed across 12 drives. EC:8 still tolerates 4 simultaneous drive failures, but it uses storage more efficiently than EC:4 because the ratio of parity shards to total shards is lower. The read quorum for EC:8 is 8 shards, and the write quorum is 9 shards.

The number of drives in an erasure set is always K+M, and MinIO imposes a maximum of 16 drives per erasure set. This limit is a design constraint that ensures erasure sets remain manageable and that the Reed-Solomon encoding and decoding operations stay computationally efficient. A deployment with more than 16 drives spans multiple erasure sets, with MinIO automatically distributing objects across erasure sets for load balancing.

Erasure sets are immutable after a cluster is deployed. Once a MinIO cluster is initialized with a given number of drives per erasure set, that configuration cannot be changed without redeploying the cluster. This immutability is intentional: the erasure set structure determines how data is distributed across drives, and changing it mid-operation would require rewriting all existing data. Organizations planning MinIO deployments should therefore choose their erasure coding configuration carefully, taking into account the expected number of drives, desired fault tolerance, and storage efficiency requirements.

Storage efficiency is one of erasure coding's key advantages over replication. Traditional three-way replication, common in systems like HDFS, stores three full copies of every object, resulting in 200% storage overhead — 1 TB of data requires 3 TB of raw storage. Erasure coding dramatically reduces this overhead. An EC:4 configuration (4 data + 4 parity shards) has 50% overhead — 1 TB of data requires 2 TB of raw storage. EC:8 (8 data + 4 parity shards) has 33% overhead — 1 TB of data requires approximately 1.5 TB of raw storage. For petabyte-scale deployments, this efficiency difference translates directly into significant cost savings on hardware and power.

MinIO's erasure coding implementation also includes self-healing capabilities. When MinIO reads an object and detects that one or more shards are corrupted or missing — due to a failed drive, a bitrot error, or a network fault during a previous write — it automatically reconstructs the missing shards using the surviving ones and writes the repaired shards back to healthy drives. This process is transparent to the reading application: the object is returned correctly, and the repair happens in the background. Over time, self-healing keeps the cluster in a fully healthy state even as individual drives degrade and fail.

Bitrot detection is part of this self-healing system. MinIO computes a checksum for every shard when it is written and verifies that checksum on every read. If a shard's data does not match its checksum, MinIO treats it as if the shard were missing and reconstructs it from the remaining shards. This protects against silent data corruption that could otherwise go undetected for months or years on spinning disk storage.

The combination of Reed-Solomon erasure coding, automatic self-healing, and bitrot detection makes MinIO's storage layer exceptionally robust. Data durability in a correctly configured MinIO cluster is rated at eleven nines — 99.999999999% — meaning the probability of permanent data loss is negligible under normal operational conditions. This level of durability, combined with storage efficiency substantially better than replication, is why erasure coding is the recommended data protection strategy for all production MinIO deployments.
