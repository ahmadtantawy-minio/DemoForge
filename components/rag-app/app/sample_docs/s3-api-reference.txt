S3 API Reference: Core Operations and Authentication

The S3 API is an HTTP-based interface for interacting with object storage. Every operation is a standard HTTP request — a PUT to write data, a GET to read it, a DELETE to remove it — authenticated with a signing scheme that proves the caller's identity without transmitting credentials in plaintext. MinIO implements the full S3 API, which means every operation described here works identically against MinIO and Amazon S3.

Authentication in the S3 API uses AWS Signature Version 4, commonly called SigV4. SigV4 is an HMAC-SHA256 signing scheme where the caller uses their secret access key to create a cryptographic signature over a canonical representation of the request — including the HTTP method, URI, query parameters, headers, and a hash of the request body. This signature is included in the Authorization header. The server recomputes the signature independently and rejects the request if the signatures do not match. Because the signature covers the request body hash, SigV4 also provides integrity protection: any modification to the request in transit will cause signature verification to fail.

PutObject is the fundamental write operation. A client sends an HTTP PUT request to the URL of the target object, with the object data in the request body. Optional headers carry metadata: Content-Type describes the data format, and custom metadata can be attached using headers prefixed with x-amz-meta-. MinIO stores this metadata alongside the object and returns it on every subsequent read. PutObject is atomic — the object either appears fully written or not at all, so readers never see partial writes. For objects up to 5 GB, a single PutObject suffices. Larger objects require the multipart upload API.

GetObject retrieves an object's data in an HTTP GET response. The full object is returned by default, but clients can request a byte range using the HTTP Range header, which causes the server to return only the specified slice of the object. Range requests are essential for streaming large media files, resuming interrupted downloads, and reading specific sections of large columnar data files without fetching the entire object. GetObject also returns all metadata associated with the object in the response headers.

ListObjectsV2 is the standard operation for enumerating objects in a bucket. Because buckets can contain billions of objects, listing is paginated: each response returns up to 1,000 keys and a continuation token that the client passes in the next request to retrieve the subsequent page. Two parameters make listing more useful in practice. The prefix parameter filters results to only keys that begin with a specified string, allowing callers to simulate directory-style listing by specifying a path prefix. The delimiter parameter, when set to a slash, causes the API to return common prefixes — logical directory names — in a separate field, rather than enumerating every object beneath them individually.

Multipart upload handles objects too large to send in a single request, or where parallel uploading is beneficial for throughput. The process has three phases. First, the client calls InitiateMultipartUpload, which returns an upload ID identifying this particular multipart session. Second, the client uploads individual parts using UploadPart, each identified by the upload ID and a part number from 1 to 10,000. Parts can be uploaded in any order and in parallel across multiple connections or hosts. Each part must be at least 5 MB except for the final part, which can be any size. Third, when all parts are uploaded, the client calls CompleteMultipartUpload with the list of part numbers and their ETags. MinIO assembles the parts into a single object and makes it available atomically. If the upload is abandoned, the client or an automated lifecycle rule can call AbortMultipartUpload to free the storage consumed by the incomplete parts.

Presigned URLs allow temporary, unauthenticated access to specific objects. A client with valid credentials calls the SDK's presign function, specifying the operation (typically GetObject or PutObject), the object key, and an expiration duration. The SDK constructs a URL that embeds all the authentication parameters as query strings, signed with the caller's credentials. Anyone who receives this URL can perform the specified operation until the URL expires, without needing their own credentials. Presigned URLs are widely used for sharing private files with end users, for browser-based direct uploads to object storage, and for generating download links in web applications.

Bucket notifications allow MinIO to publish events to external systems when objects are created, accessed, or deleted. A notification configuration on a bucket specifies which event types to publish (s3:ObjectCreated:Put, s3:ObjectRemoved:Delete, and others) and which destination to send them to. MinIO supports several notification targets including Apache Kafka, AMQP message queues, webhooks, and NATS. Applications use bucket notifications to trigger downstream processing — for example, starting a thumbnail generation job whenever a new image is uploaded, or updating a search index when documents are modified.

Object locking prevents objects from being deleted or overwritten during a retention period. Locking operates in two modes. Governance mode allows users with special permissions to override the lock, making it suitable for internal data management policies where administrators may need flexibility. Compliance mode is absolute: no user, including the root account, can delete or modify a locked object before its retention date. Compliance mode is designed for regulatory requirements such as SEC Rule 17a-4, which mandates that financial records be stored in a non-rewritable, non-erasable format.

Versioning keeps every historical version of every object in a bucket. When versioning is enabled, a PUT to an existing key does not overwrite the previous version — it creates a new version with a unique version ID, leaving all prior versions accessible. A DELETE operation on a versioned object does not remove any data; instead, it inserts a delete marker, which is a special zero-byte object that makes the key appear deleted to standard listing and GET requests. The underlying versions remain intact and can be restored by removing the delete marker. Versioning and object locking work together to provide a comprehensive data protection strategy for critical datasets that must be preserved against both accidental modification and deliberate deletion.
