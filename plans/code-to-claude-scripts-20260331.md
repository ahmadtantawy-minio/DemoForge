### demoforge.sh

Entry point for all local operations. Sources `.env.hub` and `.env.local` before `docker compose up` (as of latest commit). Key commands:
- `start` — builds component images, `docker compose build`, `docker compose up -d`
- `stop` — stops DemoForge + all demo containers
- `restart` — stop then start
- `build` — build only, don't start
- `clean` — stop + remove volumes + prune demo networks
- `nuke` — clean + remove built images
- `dev:be` — run backend locally with uvicorn (hot-reload)
- `dev:fe` — run frontend locally with Vite (hot-reload)

### minio-gcp.sh (in scripts/)

GCP management script with 4 modes:

| Mode | Flag | Description |
|------|------|-------------|
| Deploy | (none) | Fresh VM deploy — project, disk, firewall, MinIO container, Docker registry |
| Update | `--update` | In-place MinIO → AIStor upgrade (preserves data) |
| Activate | `--activate` | Register AIStor Free license via mc |
| Gateway | `--gateway` | Deploy VPC + Cloud Run gateway + hub-connector image |

Gateway mode (11 steps): verify VM → enable APIs → create VPC/subnet → migrate VM to VPC → firewall rules (internal + dev IP + IAP SSH) → VPC connector → API key gen → build/deploy Cloud Run gateway (Caddy) → verify → build hub-connector image → generate `.env.hub`

## 5. Hub Connector

The hub-connector is NOT committed as source — it's built dynamically during `minio-gcp.sh --gateway` (Step 10) in a temp directory and pushed to GCR as `gcr.io/minio-demoforge/demoforge-hub-connector:latest`.

**Architecture:** Caddy reverse proxy running as a Docker container on the FA's laptop.

| Local Port | Gateway Route | Target |
|------------|--------------|--------|
| 9000 | `X-Service: s3` header | MinIO S3 API |
| 5000 | `/v2/*` path | Docker Registry |
| 9001 | `X-Service: console` header | MinIO Console |
| 8080 | `/health` | Gateway health |

Connector injects `X-Api-Key` and `X-Service` headers on every request. Gateway validates API key, routes based on `X-Service` header (S3/console) or path (`/v2/*` for registry).

**S3 signature limitation:** S3 Signature V4 includes the `Host` header in the HMAC. Two proxy hops change the Host, breaking signatures. Workaround: template sync uses direct IP (firewall-restricted) instead of going through the connector. The connector works for unsigned operations (health checks, console UI, Docker registry).

No Caddyfile files are committed to the repo.

### Template Override System

- Override backs up original to `data/template-backups/{id}/vN.yaml`
- SHA-256 hash verification of backup integrity
- Override manifest at `data/template-backups/.override-manifest.json`
- Cards show amber "Customized" badge
- Revert available in dev mode (`DEMOFORGE_MODE=dev`)

### Sync Engine

- Uses MinIO Python SDK (`minio` package)
- Syncs from `DEMOFORGE_SYNC_ENDPOINT` / `DEMOFORGE_SYNC_BUCKET` / `DEMOFORGE_SYNC_PREFIX`
- Writes to `synced-templates/` with `.sync-manifest.json` tracking
- Auto-syncs on backend startup, manual trigger via API
- Currently syncs via direct IP (not through gateway) due to S3 signature constraints

## 8. Docker Compose

```yaml
services:
  backend:
    build: ./backend
    ports: ["9210:9210"]
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock    # Docker-in-Docker
      - ./components:/app/components:ro               # Component manifests
      - ./demos:/app/demos                            # Saved demos
      - ./data:/app/data                              # Runtime state
      - ./demo-templates:/app/demo-templates:ro        # Built-in templates
      - ./user-templates:/app/user-templates            # User templates (rw)
      - ./synced-templates:/app/synced-templates         # Synced templates (rw)
      - ./backend/app:/app/app                          # Live reload source
    environment: [see Section 4]
    labels: ["demoforge.role=backend"]

  frontend:
    build: ./frontend
    ports: ["3000:3000"]
    depends_on: [backend]
    environment:
      - VITE_API_URL=http://localhost:9210
```

**Key architectural notes:**
- Backend has Docker socket access — controls all demo containers
- Backend source is volume-mounted for live reload during development
- Frontend is fully rebuilt on `make restart` (not volume-mounted)
- No container networking between DemoForge and demos — demos get their own networks
- `demoforge.sh` sources `.env.hub` then `.env.local` before `docker compose up`
- `DEMOFORGE_REGISTRY_HOST` uses `host.docker.internal:5000` to reach hub-connector from inside backend container; Docker pulls use `localhost:5000` (via host daemon socket)

## 9. Known Constraints & Gotchas

1. **S3 signatures vs. proxy chain**: S3 SigV4 signs the `Host` header. The hub-connector → gateway → MinIO chain changes Host at each hop, breaking signatures. Template sync bypasses the gateway and uses direct IP with firewall restriction.

2. **Frontend requires rebuild**: Unlike the backend (volume-mounted), frontend changes require `make restart` or `docker compose build frontend` to take effect.

3. **Platform image tag mismatch**: Locally-built images are tagged `demoforge-backend:latest` but the registry version is `demoforge/demoforge-backend:latest`. The image status API checks both tags via alternate tag fallback.

4. **Hub-connector not committed**: The Caddyfile is generated dynamically during `--gateway` mode. To update the connector image, re-run `minio-gcp.sh --gateway` or build locally and push to GCR.

5. **Hardcoded IPs**: Several scripts still reference `34.18.90.197` (old public IP) as defaults. These are overridden by `.env.hub`/`.env.local` at runtime, but the defaults are stale.

6. **VM public IP**: The VM was moved to a private VPC but later got an ephemeral external IP added back (restricted to admin IP) for direct S3 access. This IP changes on VM restart.
